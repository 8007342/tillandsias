Files touched (Windows-owned only):
- `crates/tillandsias-windows-tray/tests/cli_integration.rs`:
  - New `status_once_json_has_all_7_keys_and_exit_code_matches` test:
    runs `tray.exe --status-once --json`, asserts exit ∈ {0, 1, 2}
    (all three are valid "report ran to completion" outcomes), parses
    stdout as JSON, asserts all 7 pinned `StatusReport` keys present
    (`reachable`, `wire_version`, `phase`, `podman_ready`,
    `last_event`, `error`, `exit_code`).
  - **Self-consistency assertion**: the `exit_code` field embedded in
    the JSON must equal the process exit code returned by `Command::
    status().code()`. Pins the invariant that `status_exit_code()`
    (the pure-helper deriving 0/2/1 from `reachable` + `phase`)
    matches the actual exit. A regression that flipped the embedded
    value out of sync with the process exit would surface here pre-
    build — operators relying on `report.exit_code` for scripting
    couldn't be silently lied to.
- `openspec/litmus-tests/litmus-windows-tray-diagnose-cli-surface.yaml`:
  - "end-to-end cli_integration tests attached" step extended to also
    require the new test name grep. Now requires 5 test names total.

**Smoke**: the test itself IS the smoke — runs the real binary, parses

**Litmus**:

- `cargo fmt`: clean.

**Findings** (free-form):
- The self-consistency assertion (`json_exit == process_exit_code`)
  is the most operationally important part of the new test. Support
  tooling reading `--status-once --json` to script "is the VM Ready?"
  uses the `exit_code` field as the canonical signal; if a future
  refactor flipped it out of sync with the actual process exit, the
  support script would silently make wrong decisions. Now caught
  pre-build.
- The 5 cli_integration tests now cover 5 of the 7 CLI modes
  end-to-end. The 2 NOT covered (`--provision-once` and
  `--status-once` human mode) are intentionally skipped:
  `--provision-once` requires WSL state + control wire interaction
  which makes it environment-dependent and flaky; `--status-once`
  human mode is redundant given the `--json` test (same code path
  with a different print).
- The new test catches a class of bugs the inline schema-pin test
  cannot: a regression in `collect_status_report` that yields
  inconsistent state (e.g. setting `reachable: true` but
  `phase: None` and not honoring `status_exit_code`'s "reachable
  but no phase → 1" branch) would silently change behavior. The
  end-to-end test verifies the WHOLE pipeline (sniff → derive →
  serialize → exit) stays internally consistent.
