## Category 1: Known Blockers (Non-Critical)

### Blocker: Tray Litmus Timeout @ 120s for Interactive Tests

**Severity**: Low (non-blocking for current phase)  
**Scope**: `litmus:tray-litmus-chain` verification workflow  
**Status**: Gated feature, not required for release  

**Details**:
- Interactive menu tests (e.g., "Attach Here" flow, container start tracking) can exceed 120s on slow CI agents
- Current: Tests are optional in `./build.sh --ci` (no auto-trigger); only enforced in manual `./build.sh --ci-full`
- Impact: Release workflows are manual (`workflow_dispatch`), so timeout is never hit in production CI
- Workaround: Increase timeout override for local development, or skip on slow agents

**Owner**: Future wave (ci-litmus-reliability)  
**Action**: Document timeout tolerance in `cheatsheets/testing/tray-litmus-timeout.md` before implementing parallel litmus runners

## Category 2: Edge Cases (Candidates for Future Hardening)

### Edge Case 1: Rapid Project Switches During Initialization

**Severity**: Low (edge case, unlikely in normal use)  
**Scope**: User clicks between projects while containers are initializing  

**Details**:
- Scenario: User clicks "Project A → Attach Here", then immediately clicks "Project B → Attach Here"
- Current behavior: Second click is blocked by `can_start_project()` → `lifecycle_state` guard (returns false during Initializing)
- Possible outcome: Menu briefly shows inconsistent state (old project's icon, new project's label)
- Mitigation: Guards prevent invalid state entry; menu rebuild catches inconsistency within ~100ms

**Candidate task**: `tray/rapid-switch-resilience`  
- Adds explicit test: `test_rapid_project_switches_rejected_during_init()`
- Adds defensive check: `projects_hash` detects menu-level inconsistency and triggers rebuild
- Owner: Future wave (tray-reliability)

---

### Edge Case 2: Cache Corruption Recovery

**Severity**: Medium (can block container start)  
**Scope**: Per-project cache (`.cache/tillandsias-project/`) becomes partially corrupted  

**Details**:
- Scenario: `/home/forge/.cache/tillandsias-project/cargo/` contains broken symlinks or truncated JSON lock files
- Current behavior: Container mounts cache, but `cargo build` may fail trying to parse corrupted metadata
- No current recovery: Cache is treated as immutable; corruption requires manual `rm -rf ~/.cache/tillandsias-project`

**Candidate task**: `tray/cache-integrity-check`  
- Add `verify_cache_layout()` function to check for broken symlinks, unreadable files
- Call before container launch in `handle_attach_here()` / `handle_maintenance()`
- If corruption detected: Log warning, offer user option to wipe cache, proceed with fresh container
- Owner: Future wave (cache-resilience)

---

### Edge Case 3: Forge Image Staleness Detection Edge

**Severity**: Low (rare, only if Containerfile manually edited without rebuild)  
**Scope**: Image cache validity when Containerfile changes but hash misses  

**Details**:
- Current: `forge_image_tag()` computes hash from `flake.nix` + `flake.lock` (catches all source changes via Nix)
- Edge: If user manually edits `images/default/Containerfile` and calls `podman build` directly, hash doesn't change
- Impact: Tray still sees old cached image; new layers not included
- Mitigation: This requires deliberate bypass of `--init` (normal path always detects changes)
- Not a blocker: No known users doing manual Containerfile edits; normal workflow is `./build.sh --release`

**Candidate task**: `tray/containerfile-hash-validation`  
- Add `Containerfile` modification time to hash computation (in addition to Nix hash)
- Detects manual edits even if `flake.nix` unchanged
- Owner: Future wave (image-staleness-hardening)

## Category 3: Platform-Specific Limitations (Known, Expected)

### Limitation 1: GTK4 Runtime Unavailable on Headless Systems

**Severity**: Low (expected behavior)  
**Scope**: Tray mode on servers without GTK4  
**Behavior**: Transparent fallback to headless  

**Details**:
- Constraint: Tray mode requires GTK4 display stack (X11 or Wayland)
- Systems affected: SSH sessions, Docker, systemd containers, CI agents, headless servers
- Current behavior: `os::detect_display()` tries GTK4, falls back to headless silently
- User impact: Zero — headless mode works identically; no manual flag required

**Not a gap**: This is intentional design (tray-first, headless-fallback)  
**No action required**: Works as specified

---

### Limitation 2: D-Bus Socket Dependency on Linux

**Severity**: Low (expected behavior, documented)  
**Scope**: Tray initialization requires active D-Bus session socket  
**Behavior**: Graceful degradation to headless  

**Details**:
- Constraint: Linux tray uses StatusNotifierItem protocol over D-Bus
- Systems affected: Systems without active user D-Bus session (e.g., SSH, isolated containers)
- Current behavior: Tray tries to connect; on failure, logs error and transitions to Error state
- Recovery: User can invoke `tillandsias --headless /path` to bypass tray entirely

**Not a gap**: This is Linux platform reality  
**No action required**: Headless fallback covers all use cases

---

### Limitation 3: Container Networking Isolation (Not Enforced Across Projects)

**Severity**: Medium (by design, not a security gap)  
**Scope**: Containers from different projects can reach each other via enclave network  

**Details**:
- Current: All forge containers connect to `tillandsias-enclave` bridge network
- Behavior: Project A forge can ping/reach Project B forge (both on same network)
- Reason: Shared proxy, git service, inference service require network connectivity
- Mitigation: Proxy allowlist restricts external routes; internal cross-project traffic is allowed (feature, not bug)

**Not a gap**: This is intentional enclave design  
**By-design**: Enclave is collaborative, not adversarial; agents trusted within same enclave  
**No action required**: No change needed for current phase

## Category 4: Performance Gaps (Candidates for Future Optimization)

### Gap 1: Sequential Image Builds in Init Command

**Severity**: Low (affects startup time only)  
**Scope**: `tillandsias --init` builds six images sequentially  
**Impact**: Cold-start init can take 5-10 minutes (all layers downloaded, compiled)  

**Details**:
- Current: Images built in order: proxy → forge → git → inference → chromium-core → chromium-framework
- Reason: Dockerfile layer caching requires sequential build; later images may depend on earlier ones
- Reality: Nix-based builds (via `flake.nix`) can parallelize, but `podman build` is serialized
- Mitigation: Docker layer caching makes subsequent inits fast (~30s)

**Candidate task**: `tray/parallel-image-builds`  
- Investigate `buildx` with multiple builders for independent images (proxy, inference, chromium-* don't depend on forge)
- Measure: Parallel vs. sequential build times
- Owner: Future wave (build-performance)  
**Note**: Not blocking; cold-start is rare (first-time users only)

---

### Gap 2: Cache Rebuild Time on Project First-Access

**Severity**: Low (affects startup time only)  
**Scope**: First `cargo build`, `npm install`, etc. in new forge container  
**Impact**: "Building environment..." chip may persist for 30-60s on large Rust/Node projects  

**Details**:
- Current: `forge_available` flag gates "Attach Here" until image is ready (binary is built and accessible)
- Scenario: User clicks "Attach Here" on a new Rust project with 500+ transitive dependencies
- Impact: First `cargo build` downloads crates.io index + all crates (can be 500 MB)
- Mitigation: Per-project cache persists across container restarts; second launch is fast

**Candidate task**: `tray/shared-cache-prewarming`  
- Option 1: Pre-pull top-100 crates (serde, tokio, etc.) on tray startup
- Option 2: Parallel download of common registry indexes in background
- Owner: Future wave (cache-prewarming)  
**Note**: Not blocking; acceptable startup latency for new projects

---

### Gap 3: Project List Discovery Performance

**Severity**: Very Low (affects startup time only)  
**Scope**: Project discovery via `discover_projects()` does full directory scan  
**Impact**: ~100ms on fast SSDs; can be longer on network mounts  

**Details**:
- Current: `discover_projects()` walks `~/src/` directory on tray startup
- Cached: Results stored in `TrayUiState::projects`; menu rebuilt only on hash change
- Scenario: User with 1000+ projects in `~/src/` (unlikely, but possible)
- Impact: First tray startup would scan all directories

**Candidate task**: `tray/lazy-project-discovery`  
- Replace full scan with inotify-based watch on `~/src/`
- Populate menu incrementally; show first 20 projects immediately, load rest in background
- Owner: Future wave (startup-performance)  
**Note**: Very low priority; current latency is imperceptible for normal use
