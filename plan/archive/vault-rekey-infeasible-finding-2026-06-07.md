## 1. Headline: literal rekey is impossible (high confidence, cited)

Stock Vault **cannot** make the active Shamir unseal share equal a host-chosen value K
(= `HKDF(machine-id ‖ installation-uuid)`):

- `vault operator init` / `POST /sys/init` accept only `secret_shares`, `secret_threshold`,
  `pgp_keys`. `pgp_keys` only *encrypts the generated output shares*; Vault generates the key.
- `vault operator rekey` / `/sys/rekey`: the `[KEY]`/`key` parameter *submits existing* unseal
  keys to reach quorum; the new shares are *randomly generated by Vault* (optionally
  pgp-encrypted / gated by `verification_required`). No parameter installs a chosen value.

Citations (official): developer.hashicorp.com `/vault/docs/commands/operator/init`,
`/vault/api-docs/system/init`, `/vault/docs/commands/operator/rekey`,
`/vault/api-docs/system/rekey`, `/vault/docs/concepts/seal`. (10+ URLs in the raw output.)

⇒ **Step 32 exit criterion 1 ("installs the host HKDF key via `vault operator rekey`") is
unachievable. The spec mandates an impossible mechanism and MUST be refined before any
implementation can satisfy it.** Note the spec is already self-inconsistent: lines 73-76,
97-101, 103-107 describe the feasible *keychain-held-key* model, while lines 84-86 / 93 claim
that key is "installed as the actual Shamir share via `vault operator rekey`."

Corollary research facts (also cited, high confidence):
- The **root token is needed only ONCE** (first-boot setup); auth methods / engines / policies
  persist behind the barrier across restarts. Unseal needs only the unseal key, never root.
- **Deleting the root token after handover is HashiCorp best practice** and reversible via
  `operator generate-root` (gated on a quorum of unseal keys, not the old root token) — *as long
  as the unseal key remains recoverable.*

## 2. Confirmed brick bug (independent of the rekey question)

`images/vault/entrypoint.sh:167` runs `ROOT_TOKEN="$(cat /vault/data/root.token)"` on **every**
subsequent boot, but the token is written only on first boot (`:151`) and **deleted** by the
host after one-time handover (`vault_bootstrap.rs:709-717`). After handover the file is gone
forever ⇒ under `set -euo pipefail` (`:39`) the failed `cat` aborts the entrypoint before unseal
⇒ a container recreate (which `ensure_vault_running` does on the *same* volume) **bricks Vault**.
The `:167` read is also functionally unnecessary for unseal (`:176-180` uses the unseal key).

## 3. Feasible mechanism (designed, NOT yet adversarially verified)

"Keychain-held Vault-generated Shamir share" — the unique option satisfying every spec invariant
without a literal rekey (transit auto-unseal needs a *second* unsealed Vault → rejects
"self-contained"; the XOR envelope is spec-forbidden):

1. First boot: Vault generates the random single Shamir share (keep the `init` call).
2. The **host captures that generated share once** (exactly like the root-token handover today),
   stores it in the host OS keychain under a new versioned user `vault-shamir-share-v1`, then
   deletes the on-disk copy. The keychain is the host-binding facility.
3. Every boot: the host feeds the keychain-held share into the existing `tillandsias-vault-unseal`
   podman secret (tmpfs `/run/secrets`); the entrypoint POSTs it to `/v1/sys/unseal`. No
   envelope, no `init.json`, no `root.token` on `/vault`, no rekey, no second Vault.
4. Handover artifacts live ONLY on tmpfs (`/run/vault-handover/...`), consumed+deleted by the
   host. Policy/auth/audit convergence is gated to **first boot only** (root token in-memory),
   so subsequent boots never need a root token. Encoding: **base64 end-to-end** (Vault returns
   `keys_base64`).

K (HKDF) becomes optional defense-in-depth, not the unseal authority (you can't make share==K).

## 5. Implementation plan (designed, NEEDS adversarial review + e2e before landing)

Concrete remove/add lists for `entrypoint.sh`, `vault_bootstrap.rs`, and a new instant
structural guard `litmus-vault-entrypoint-hardened-shape.yaml` are in the raw workflow output
(§ entrypoint_changes / vault_bootstrap_changes / guard_changes). **Brick risks** the next cycle
MUST verify (from the synthesis risk list):
1. Handover ordering — capture the share inside `wait_for_vault_ready` on the first healthy
   boot, BEFORE returning; never delete the tmpfs copy from the entrypoint side; fail-closed if
   the keychain write errors.
2. Encoding mismatch — base64 host↔entrypoint or unseal silently fails (sealed stays true).
3. First-boot chicken-and-egg — keychain has no share yet; first boot must unseal from the
   in-memory generated share and ignore the filler podman secret (gate strictly on
   `INITIALIZED != true`).
4. Subsequent-boot root token — policy/auth/audit MUST be gated first-boot-only or it bricks.
5. Existing POC volumes lose the envelope recovery path on upgrade → document
   `podman volume rm tillandsias-vault-data` re-bootstrap (POC).
6. Keychain availability (headless/CI no Secret Service) — pre-existing dependency.

e2e gate (must pass before "done"): first-boot bring-up with **no prompt**; `/vault/data` has no
`init.json`/`init.envelope`/`root.token`; share present in keychain; **container recreate
