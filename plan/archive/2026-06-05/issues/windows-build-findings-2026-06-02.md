**Findings** (free-form):
- The 5-touchpoint discipline works as designed. The exact-count
  assertion in particular was the mechanical lock that forced the
  cheatsheet + cli_integration test + litmus pin to all bump in
  lockstep. Without it, I could have shipped the field with stale
  documentation.
- The pre-computed `exit_code` field is operationally useful for
  pipeline scenarios: `if (tray.exe --diagnose --json | jq -e
  '.exit_code == 0') { ... }` works cleanly. Today the same check
  required capturing the process exit code separately.
- Order in the JSON: I placed the field right after `install_path`
  (next to other binary-identity fields). Alphabetical serde order
  would place it differently, but since `serde::Serialize` derives
  preserve declaration order for structs, the JSON shape is now:
  `{version, build_commit, install_path, exit_code, log_path, ...,
  wire, recent_log_tail}`. The cheatsheet schema block matches this
  order.
- The 17-key shape now naturally splits into:
  - Binary identity: `version`, `build_commit`, `install_path`
  - **Verdict: `exit_code` (NEW)**
  - Logs: `log_path`, `log_exists`, `log_size_bytes`
  - Host software: `wsl_version`, `os_version`, `wt_present`
  - WSL distro + rootfs: `distro`, `distro_registered`,
    `distro_running`, `release_tag`, `manifest_pin_x86_64_tar`
  - Wire: `wire{}` (5 sub-fields)
  - Recent activity: `recent_log_tail`
- The print_human sectioned output (5 visual groups from 2 ticks ago)
  is now slightly out of sync with the JSON — exit_code lives at
  position 4 in JSON but isn't displayed as a row in human mode
  (the footer already displays it as `Status: ... (exit N)`).
  Discrepancy is fine: the human mode renders the verdict via the
  Status footer; the JSON exposes it as a parseable field.
