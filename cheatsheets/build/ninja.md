- **Last updated:** 2026-04-25

**Version baseline**: ninja 1.12+ (Fedora 43 `ninja-build`).
**Use when**: faster make-replacement build orchestrator. Almost always generated by CMake/Meson, rarely hand-written.

## Quick reference

| Command | Effect |
|---|---|
| `ninja` | Build the default target in the current dir's `build.ninja` |
| `ninja <target>` | Build a specific named target |
| `ninja -C <dir>` | Run as if invoked from `<dir>` (parallels `make -C`) |
| `ninja -j <N>` | Override parallelism (default = auto-detected, ~CPUs+2) |
| `ninja -k <N>` | Keep going until `<N>` jobs fail (`-k 0` = unlimited) |
| `ninja -v` | Echo full command lines (not just rule names) |
| `ninja -n` | Dry-run; print what would build, do nothing |
| `ninja -t targets` | List every target known to `build.ninja` |
| `ninja -t targets all` | List all output paths declared by build edges |
| `ninja -t clean` | Remove all built outputs (graph-aware, unlike `rm -rf build/`) |
| `ninja -t graph <tgt>` | Emit Graphviz `.dot` of the dep graph for `<tgt>` |
| `ninja -t commands <tgt>` | Print the shell commands needed to build `<tgt>` |
| `ninja -t deps` | Show recorded header deps (from `depfile`/`deps = gcc`) |
| `ninja --version` | Print version (Meson/CMake require ≥ specific minimums) |

Rule/build/variable syntax in `build.ninja`:

```ninja
cflags = -O2 -Wall

rule cc
  command = gcc $cflags -MMD -MF $out.d -c $in -o $out
  depfile = $out.d
  deps    = gcc

build foo.o: cc foo.c
build foo:   link foo.o bar.o
default foo
```

## Common patterns

### Pattern 1 — CMake → Ninja (most common path)

```bash
cmake -S . -B build -G Ninja -DCMAKE_BUILD_TYPE=Release
ninja -C build                  # build
ninja -C build install          # if `install` target exists
ninja -C build -t targets       # discover what is buildable
```

### Pattern 2 — Meson → Ninja (Meson's only backend)

```bash
meson setup build
meson compile -C build          # wraps `ninja -C build`
ninja -C build test             # run tests directly
```

### Pattern 3 — Capture `compile_commands.json` for clangd / IDEs

```bash
ninja -t compdb cc cxx > compile_commands.json
# CMake equivalent: -DCMAKE_EXPORT_COMPILE_COMMANDS=ON writes it automatically
# Or use `bear -- ninja` if the generator does not cooperate
```

### Pattern 4 — Inspect why something rebuilds

```bash
ninja -d explain <target>       # Ninja prints the reason for every dirty edge
ninja -d stats                  # Cache hit/miss, edge counts, rss usage
```

### Pattern 5 — Parallel + stop-on-first-error iteration

```bash
ninja -j $(nproc) -k 1          # default fail-fast, explicit thread count
ninja -j 1                      # serial, when chasing race-y build bugs
```

## Common pitfalls

- **`build.ninja` is a generated artefact** — hand-editing works but is throwaway: the next `cmake`/`meson` regen wipes it. If you need persistent build logic, change the generator's input (`CMakeLists.txt`, `meson.build`), not `build.ninja`.
- **`$` is Ninja's escape, not Bash's** — `$var` references a Ninja variable; `$$` is a literal `$` in a `command =` line. Forgetting the doubling means your shell sees an empty string and the rule silently misbehaves.
- **Rule definition changes do NOT auto-rerun** — Ninja hashes the *command line* and rebuilds when it changes, but the generator (CMake/Meson) is what regenerates `build.ninja`. Edits to source files trigger rebuilds; edits to a `CMakeLists.txt` trigger the configure step first (CMake auto-reconfigures via the `build.ninja` rebuild edge).
- **Header dependencies require `depfile` + `deps`** — without `deps = gcc` (or `msvc`) and a `-MMD`/`/showIncludes` flag in the command, edits to a `.h` won't rebuild dependents. CMake/Meson handle this; hand-written rules forget it constantly.
- **Missing `mkdir` for output directories** — Ninja does NOT create parent dirs of `$out`. If a rule writes `obj/foo.o` and `obj/` doesn't exist, the command fails. Either prepend `mkdir -p $$(dirname $out) &&` or rely on a generator that emits implicit dir-creation edges.
- **`ninja -t clean` is graph-aware, `rm -rf build/` is not** — `-t clean` only deletes things Ninja knows it built. After switching branches, a stale generated header from the *other* branch survives `-t clean`. Use `git clean -fdx build/` or fully reconfigure for branch hops.
- **`default` directive controls bare `ninja`** — without an explicit `default <target>`, plain `ninja` builds *every* declared target. CMake sets a sensible default (`all`); hand-written ninja files often don't, leading to surprise full builds.

## When to choose Ninja vs Make

Ninja is faster (parallel scheduler, smaller per-edge overhead) and far simpler to parse — that's why CMake, Meson, and gn all generate it. Pick Ninja whenever your generator supports it. Hand-written `Makefile`s still earn their keep at the *top* level (orchestration, phony targets, "run tests then deploy") because Make is everywhere and humans actually read it; nobody hand-writes `build.ninja`.
