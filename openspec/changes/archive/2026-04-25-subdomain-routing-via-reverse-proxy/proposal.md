## Why

When a coding agent inside the forge launches a dev server (e.g.,
`flutter run -d web-server`, `vite`, `next dev`, `opencode serve`), it
prints something like `Server running at http://localhost:8080`. That
URL is meaningless to the human:

- It refers to localhost *inside* the forge container ŌĆö not reachable
  from the host browser.
- The port is whatever the agent picked, often colliding across
  projects.
- Multiple concurrent forges all advertise port 8080.
- If we publish the port to the host with `-p 8080:8080`, the server
  becomes reachable from the host's LAN ŌĆö which violates Tillandsias's
  enclave security model.

We already have a partial answer for opencode-web: the tray allocates a
loopback-only host port and the user opens `<project>.localhost:<port>`.
But that scheme breaks down for arbitrary agent-spawned dev servers
because the tray doesn't know about them, the port is unstable, and the
port-numbered URL is ugly.

This proposal generalises to a single, stable convention for every web
service spawned anywhere in the enclave:

```
http://<project>.<service>.localhost/
```

- `<project>` ŌĆö sanitized project name (e.g., `lakanoa`, `java`).
- `<service>` ŌĆö the service identifier (e.g., `opencode`, `flutter`,
  `vite`, `next`, `storybook`).
- Port `80` always implicit ŌĆö the user never types a port number.

A new reverse-proxy listener sits at `127.0.0.1:80` on the host (host
side) and at `proxy:80` on the enclave network (forge side). It looks
up `<project>.<service>.localhost` in a per-attach routing table and
forwards to the right container at the right internal port.

Crucially:

- The reverse proxy binds to **127.0.0.1 only** on the host. `*.localhost`
  is already loopback-hardcoded by Chromium/Firefox/systemd-resolved
  per RFC 6761, so external access is impossible by construction.
- Forge agents reach `<project>.<service>.localhost` through their
  existing `HTTP_PROXY=http://proxy:3128` ŌĆö the forward proxy
  recognises the `.localhost` TLD and forwards to the reverse-proxy
  sibling at `proxy:80`. Same network stack, same CA chain we already
  control, no new external-facing surface.
- Container service ports stay internal. `flutter run` binds
  `0.0.0.0:8080` *inside the container*; the host never sees that port.

This is the missing piece that lets the enclave behave like a private
LAN with friendly DNS while preserving zero-external-exposure.

## What Changes

### Reverse-proxy listener

- New container `tillandsias-router` (or merged listener inside the
  existing `tillandsias-proxy` Squid). MVP: separate container running
  Caddy 2.x because Caddy's hostname-glob matching + on-config-change
  reload is operationally simplest.
- Two binds: `127.0.0.1:80` on host, `router:80` on enclave.
- Routing config generated by the tray at attach time and reloaded
  via Caddy admin API.

### Routing table

The tray maintains a dynamic mapping in
`$XDG_RUNTIME_DIR/tillandsias/router/Caddyfile` (tmpfs, regenerated on
each attach):

```
*.opencode.localhost:80 {
    reverse_proxy /* tillandsias-{labels.0}-aeranthos:4096
}
*.flutter.localhost:80 {
    reverse_proxy /* tillandsias-{labels.0}-aeranthos:8080
}
```

Service-port conventions:

| Service identifier | Internal port | Notes |
|--------------------|---------------|-------|
| `opencode`         | 4096          | OpenCode Web (already exists) |
| `flutter`          | 8080          | `flutter run -d web-server --web-port 8080 --web-hostname 0.0.0.0` |
| `vite`             | 5173          | Vite default |
| `next`             | 3000          | Next.js default |
| `storybook`        | 6006          | Storybook default |
| `webpack` / `wds`  | 8080          | webpack-dev-server default |
| `jupyter`          | 8888          | Jupyter notebook |
| `streamlit`        | 8501          | Streamlit |

Agents are told (via instructions/web-services.md) which port to bind
for each service; the router knows which port to forward to.

### Forward-proxy update

Squid's allowlist gains `*.localhost`, and a `cache_peer router parent
80 0` directive forwards those domains to the router. From the agent's
perspective, `curl http://myproject.flutter.localhost/` works
identically to `curl https://github.com/...` ŌĆö both go via
`HTTP_PROXY`, but `.localhost` URLs land at the enclave-internal
router instead of the external internet.

### Browser launch URL

When the tray opens a browser for OpenCode Web, the URL becomes
`http://<project>.opencode.localhost/` (no port, no path). The legacy
form `<project>.localhost:<port>` is removed.

### Host-side binding hardening

- OpenCode Web and any service the tray exposes binds **only** to
  loopback (`127.0.0.1:80` via the router). The previous random-port
  publish (`-p 127.0.0.1:<port>:4096`) is removed ŌĆö the router handles
  host-side access.
- The router itself binds to `127.0.0.1:80` only. `0.0.0.0:80` would be
  a regression and is forbidden in the spec.

### Agent instructions

`config-overlay/opencode/instructions/web-services.md` (new) tells the
agent:

1. When asked to run a web server, bind it to `0.0.0.0:<known-port>`
   inside the forge ŌĆö never `localhost:N` (the agent's localhost is
   *its container's* loopback, not the host's).
2. The URL the human will use is `http://<project>.<service>.localhost/`
   on port 80 ŌĆö never include a port number.
3. Don't try to publish ports ŌĆö the router does that.
4. To self-test from inside the forge, the agent uses `curl
   http://<project>.<service>.localhost/` (which goes through the
   forward proxy and back via the router).

## Capabilities

### New Capabilities

- `subdomain-routing-via-reverse-proxy` ŌĆö the router container, the
  port conventions, the routing-table generator, the agent instructions.

### Modified Capabilities

- `proxy-container` ŌĆö adds the `cache_peer router` directive and the
  `*.localhost` allowlist entry.
- `opencode-web-session` ŌĆö the URL format changes from
  `<project>.localhost:<port>` to `<project>.opencode.localhost`. The
  port is removed.

## Impact

- **Containers**: one new, `tillandsias-router` (Caddy 2.x, ~50MB).
- **Rust**: new module `crate::router` to manage the Caddyfile +
  reload trigger; small touchups in `handlers::ensure_proxy_running`
  and the opencode-web launcher.
- **Shell**: agent instructions file added.
- **No host network changes** ŌĆö everything stays on loopback.
- **Backwards compatibility**: the old `<project>.localhost:<port>` URL
  stops working; users get a stable `.opencode.localhost` instead.

## Implementation phasing

Given the scope, this proposal allows incremental landing:

1. **Phase 1 (this commit)**: agent instructions + spec. Agents start
   using the right hostname pattern even before the router exists.
   Server URLs they print won't resolve yet, but the convention is
   right.
2. **Phase 2**: Router container + Caddyfile generator + Squid
   `cache_peer`. OpenCode Web migrates to the new URL.
3. **Phase 3**: Service-specific instructions for Flutter / Vite /
   Next / etc. with port conventions.
4. **Phase 4**: Removal of legacy `<project>.localhost:<port>` URLs.
