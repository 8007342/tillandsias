## ADDED Requirements

### Requirement: Trace index script
The project SHALL provide `scripts/generate-traces.sh` that scans the codebase for `@trace spec:<name>` annotations and generates a navigable index.

#### Scenario: Scan scope
- **WHEN** `generate-traces.sh` is run from the repository root
- **THEN** it SHALL scan all `.rs`, `.sh`, `.toml`, and `.nix` files under the repository root
- **AND** it SHALL extract every `@trace spec:<name>` reference, capturing the file path, line number, and spec name
- **AND** sub-path references (`spec:podman-orchestration/security-hardened-defaults`) SHALL be normalized to their root spec name (`podman-orchestration`) for index lookup

#### Scenario: Spec resolution
- **WHEN** a unique spec name is found
- **THEN** the script SHALL look for the spec file at `openspec/specs/<name>/spec.md` first
- **AND** if not found there, SHALL search `openspec/changes/archive/*/specs/<name>/spec.md`
- **AND** if still not found, SHALL mark the spec as `(not found)` in the output without failing

#### Scenario: No annotations found
- **WHEN** the scan finds zero `@trace` annotations
- **THEN** the script SHALL generate a `TRACES.md` with an empty table and a note that no traces were found
- **AND** the script SHALL exit 0

### Requirement: Root TRACES.md
The repository root SHALL contain a `TRACES.md` file generated by `generate-traces.sh` that maps every unique spec name to its spec file and all annotated source locations.

#### Scenario: Table structure
- **WHEN** `TRACES.md` is opened on GitHub
- **THEN** it SHALL render a Markdown table with columns: Trace, Spec, Source Files
- **AND** each row SHALL correspond to one unique spec name from the codebase

#### Scenario: Spec link
- **WHEN** a spec has an active spec file at `openspec/specs/<name>/spec.md`
- **THEN** the Spec column SHALL contain a relative Markdown link to that file
- **AND** the link SHALL be clickable on GitHub with no plugins required

#### Scenario: Archived spec link
- **WHEN** a spec is found only in the archive
- **THEN** the Spec column SHALL link to `openspec/changes/archive/<date-name>/specs/<name>/spec.md`
- **AND** the link SHALL be prefixed with `(archived)` to signal non-active status

#### Scenario: Source file links
- **WHEN** a spec name appears in multiple source locations
- **THEN** the Source Files column SHALL list each location as a relative link with a `#L<n>` anchor
- **AND** links SHALL be relative to the repository root so they resolve correctly on GitHub

#### Scenario: Regeneration notice
- **WHEN** a reader opens `TRACES.md`
- **THEN** the file SHALL begin with a header stating it is auto-generated and how to regenerate it

### Requirement: Per-spec back-links
For each active spec that has at least one `@trace` annotation in the codebase, `generate-traces.sh` SHALL generate an `openspec/specs/<name>/TRACES.md` companion file.

#### Scenario: Back-link content
- **WHEN** `openspec/specs/<name>/TRACES.md` is opened
- **THEN** it SHALL list every source file location that traces to this spec
- **AND** links SHALL be relative from the spec directory so they resolve on GitHub

#### Scenario: No back-links for archived specs
- **WHEN** a spec is found only in the archive
- **THEN** `generate-traces.sh` SHALL NOT create a `TRACES.md` inside the archive entry
- **AND** the archive entry SHALL remain read-only from the script's perspective

#### Scenario: Spec file not modified
- **WHEN** `generate-traces.sh` runs
- **THEN** it SHALL NOT modify any `spec.md` file
- **AND** it SHALL ONLY write `TRACES.md` files

### Requirement: Build integration
`build.sh` SHALL invoke `generate-traces.sh` on every build that is not a test-only or check-only invocation, so `TRACES.md` stays current without a separate manual step.

#### Scenario: Auto-regeneration on build
- **WHEN** a developer runs `./build.sh` or `./build.sh --release`
- **THEN** `TRACES.md` SHALL be regenerated before the build completes
- **AND** the regeneration failure SHALL be non-fatal (uses `|| true` to avoid blocking builds)

#### Scenario: Skipped on test/check
- **WHEN** a developer runs `./build.sh --test` or `./build.sh --check`
- **THEN** `generate-traces.sh` SHALL NOT run
- **AND** the existing `TRACES.md` SHALL remain unchanged

### Requirement: No external dependencies
`generate-traces.sh` SHALL use only POSIX shell utilities (`grep`, `find`, `sort`, `awk`, `sed`) and SHALL NOT require any external tools, language runtimes, or network access.

#### Scenario: Toolbox-free execution
- **WHEN** `generate-traces.sh` is run on the host (outside the tillandsias toolbox)
- **THEN** it SHALL complete successfully using only tools available on a standard Fedora Silverblue host
