# spec: project-bootstrap-readme

## Status

active

## Overview

Define the automated README.md generation and validation system for Tillandsias-managed projects. Every project's README MUST be auto-generated from authoritative sources (git history, Containerfile manifests, project metadata) and validated for structure, freshness, and consistency. This ensures project documentation is always accurate and maintainable.

@trace spec:project-bootstrap-readme

## Requirements

### Requirement: README MUST have two sections: FOR HUMANS and FOR ROBOTS

Every project README MUST be structured with two distinct sections, each targeting its intended audience.

#### Scenario: Human-readable section
- **WHEN** a developer opens the project README
- **THEN** they see a FOR HUMANS section with:
  - Plain-language project description
  - Quick-start instructions (build, test, run)
  - Development workflow overview
  - Links to additional documentation
- **AND** the language is accessible to non-expert developers

#### Scenario: Machine-readable section
- **WHEN** a tool or agent parses the README
- **THEN** they can extract machine-readable metadata from the FOR ROBOTS section
- **AND** metadata includes: project type, languages, recent commits, OpenSpec items, build status
- **AND** this section MAY be YAML, JSON, or structured markdown

### Requirement: README MUST be auto-generated from source manifests

The README MUST be generated by reading authoritative sources, not hand-edited.

#### Scenario: Manifest summarizers extract project information
- **WHEN** `scripts/regenerate-readme.sh` is executed
- **THEN** it invokes language-specific summarizers:
  - `summarize-cargo.sh` — Rust projects (reads Cargo.toml)
  - `summarize-go.sh` — Go projects (reads go.mod)
  - `summarize-package-json.sh` — Node.js projects (reads package.json)
  - `summarize-maven.sh` — Java/Maven projects (reads pom.xml)
  - `summarize-gradle.sh` — Java/Gradle projects (reads build.gradle)
  - `summarize-flutter.sh` — Flutter projects (reads pubspec.yaml)
- **AND** each summarizer emits structured data (name, version, description, main entry point, dependencies)

#### Scenario: README regeneration is deterministic
- **WHEN** `scripts/regenerate-readme.sh` runs on the same project twice
- **THEN** the output is byte-identical (same formatting, same metadata order)

### Requirement: README validation ensures structure and freshness

A validator MUST check that every README conforms to the two-section structure and has current metadata.

#### Scenario: Validator confirms structure
- **WHEN** `scripts/check-readme-discipline.sh` is executed
- **THEN** it confirms:
  - README.md exists
  - FOR HUMANS section is present
  - FOR ROBOTS section is present
  - YAML front-matter is well-formed (if present)
  - Last-updated timestamp is recent (threshold: project-configurable)

#### Scenario: Validator detects staleness
- **WHEN** the last-updated timestamp in the README is older than the threshold (e.g., 7 days)
- **THEN** the validator warns: "README may be stale; run `scripts/regenerate-readme.sh`"
- **AND** the check is non-blocking (warning, not error)

### Requirement: Pre-push hook auto-regenerates README

A git pre-push hook MUST regenerate the README before allowing a push to the remote repository.

#### Scenario: Hook runs before push
- **WHEN** a developer runs `git push`
- **THEN** the hook `scripts/install-readme-pre-push-hook.sh` (if installed) executes
- **AND** it runs `scripts/regenerate-readme.sh` if the README is stale
- **AND** it stages the updated README automatically
- **AND** if the hook finds changes, it aborts the push with a message: "README updated; please review and push again"

#### Scenario: Hook installation is explicit
- **WHEN** a developer runs `scripts/install-readme-pre-push-hook.sh`
- **THEN** the hook is installed to `.git/hooks/pre-push`
- **AND** it is tracked in git (via a wrapper that installs on clone)

### Requirement: Traces ledger records agent observations

An append-only ledger file MUST track agent observations about the project state.

#### Scenario: Traces ledger location
- **WHEN** a project is initialized
- **THEN** `.tillandsias/readme.traces` is created as an append-only JSONL file
- **AND** it is committed to git (cross-machine synchronization)

#### Scenario: Agent appends observations
- **WHEN** an agent discovers project metadata or completes a bootstrap task
- **THEN** it appends a JSON record: `{"timestamp": "...", "source": "...", "event": "...", "data": {...}}`
- **AND** the record includes: agent name, observation timestamp, project state hash, manifest summary

#### Scenario: Traces ledger is readable by subsequent agents
- **WHEN** a new agent clones the project and reads `.tillandsias/readme.traces`
- **THEN** it can reconstruct the project bootstrap history
- **AND** it can detect if critical state has changed since the last observation

### Requirement: Bootstrap skills guide workflow

Four CLI skills MUST guide users through project setup and maintenance.

#### Scenario: `/startup` entrypoint detects project state
- **WHEN** a user runs `/startup` in an empty or new project
- **THEN** the skill detects the project state (empty, partially initialized, or ready)
- **AND** routes to the appropriate next step: empty-project welcome, repair flow, or status view

#### Scenario: `/bootstrap-readme` regenerates README
- **WHEN** a user runs `/bootstrap-readme`
- **THEN** the skill executes `scripts/regenerate-readme.sh`
- **AND** validates the output with `scripts/check-readme-discipline.sh`
- **AND** reports success or issues found

#### Scenario: `/status` shows project health
- **WHEN** a user runs `/status`
- **THEN** the skill outputs:
  - Recent commits (last 5)
  - OpenSpec items (active/completed)
  - README validation results
  - Traces ledger tail (last 5 entries)

## Litmus Tests

### Test: FOR HUMANS / FOR ROBOTS structure validation
- **Setup**: Run `/startup` on empty project, allow bootstrap, generate README
- **Action**: Parse README.md and check structure
- **Signal**: Two top-level sections present: `## FOR HUMANS` and `## FOR ROBOTS`
- **Pass**: Both sections present, human section readable (plain language, no YAML), robot section parseable (YAML/JSON valid)
- **Fail**: Missing sections, malformed YAML, or content in wrong section

### Test: Manifest detection and summarizer invocation
- **Setup**: Create test projects: Rust (Cargo.toml), Node (package.json), Go (go.mod)
- **Action**: Run `scripts/regenerate-readme.sh` in each
- **Signal**: README generated with project-specific content
- **Pass**: Cargo summarizer invoked for Rust, package.json summarizer for Node, go.mod for Go; content matches manifest
- **Fail**: Wrong summarizer invoked, content missing, or determinism failure (different output on re-run)

### Test: README staleness detection
- **Setup**: Create README with old timestamp (e.g., 30 days old)
- **Action**: Run `scripts/check-readme-discipline.sh`
- **Signal**: Validator outputs staleness warning
- **Pass**: Warning emitted; suggestion to run regenerate; check returns exit code 0 (non-blocking)
- **Fail**: No warning, exit code 1 (blocking), or false negative on fresh README

### Test: Pre-push hook regeneration
- **Setup**: Modify project manifest (e.g., add dependency to Cargo.toml), don't regenerate README
- **Action**: Stage changes, run `git push` (or simulate with hook invocation)
- **Signal**: Hook runs and updates README automatically
- **Pass**: README regenerated, staged, push aborted with message; user re-runs push and succeeds
- **Fail**: Hook skipped, push proceeds with stale README, or regeneration produces wrong content

### Test: Traces ledger persistence and cross-machine sync
- **Setup**: Initialize project, run bootstrap, append observations to `.tillandsias/readme.traces`
- **Action**: Commit and push to remote; clone repo on another machine
- **Signal**: `.tillandsias/readme.traces` present and contains original observations
- **Pass**: Ledger entries readable by downstream agents; history complete; format is valid JSONL
- **Fail**: Ledger missing after clone, entries corrupted, or agent cannot parse
